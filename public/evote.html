Project Overview

E-Vote is a secure online voting platform designed for small to medium organisations that need remote elections with strong guarantees of voter anonymity, integrity, and verifiable audit trails. The platform is built to production-grade standards, demonstrating end-to-end ownership of architecture, security, DevOps automation, and operational observability.

The system is decomposed into three independent Python FastAPI microservices (auth, voting, audit), a React frontend, Apache Kafka for event streaming between services, and a Kong API Gateway as the single ingress point. Each service owns its own PostgreSQL database, enforcing strict data isolation at the service boundary.

Security Design

Security is a first-class design constraint rather than an afterthought. All authentication flows use JWT tokens with a signing key stored exclusively as a Kubernetes Secret, never in source code or Docker images. Vote payloads are signed with a separate VOTE_SIGNING_KEY to guarantee tamper-evidence at the ballot level.

Kong API Gateway centralises cross-cutting security concerns: JWT validation, CORS enforcement, and rate limiting (100 req/min) are applied at the gateway layer. After Kong was introduced, duplicate JWT/CORS handling was explicitly removed from the FastAPI services (KAN-64) to eliminate a false security model where direct-to-service requests could bypass the gateway entirely.

Kubernetes NetworkPolicy resources lock backend services from direct external access, ensuring all traffic flows through Kong. A Trivy container image vulnerability scan is included as a CI quality gate, blocking merges if high-severity CVEs are detected. The enforced secrets pattern is: GitHub Actions Secrets → pipeline creates K8s Secret → pods consume via secretRef.

Microservices Architecture

The system follows a bounded-context model where each service owns its data and communicates asynchronously. The auth service handles voter registration and JWT issuance. The voting service records ballots and publishes events to Kafka. The audit service consumes those events and maintains an immutable audit log, providing a verifiable trail without coupling directly to the voting service.

Kafka decouples the voting and audit services, providing durability: if the audit service is killed mid-vote, Kafka buffers the events and replays them on restart. Each service has a minimum of three unit tests (vote cast, token generation, audit insert) enforced as a CI quality gate.

DevOps and GitOps Pipeline

Infrastructure is fully automated. Ansible playbooks provision cloud VMs with common, docker, github-runner, and k3s roles. Three self-hosted GitHub Actions runners (build-env, test-env, stage-env) are registered with custom labels and routed to the correct pipeline jobs. No secrets are baked into images; all are injected at runtime via GitHub Actions Secrets into Kubernetes Secrets.

The deployment model is GitOps: on a successful CI run, the pipeline pushes tagged images to GHCR and commits updated image tags to a separate GitOps repository. ArgoCD with auto-sync, prune, and self-heal watches that repo and reconciles the cluster state automatically. This eliminates SSH-based deploys, provides drift detection, and produces a full deployment audit trail in git.

Kustomize overlays (overlays/stage and overlays/prod) provide per-environment image tags and replica counts, keeping base manifests DRY. The CI pipeline includes a quality gate: any failing test or Trivy finding blocks the merge to main.

Key Design Decisions

Kong over NGINX: The original design used an NGINX api-gateway deployment as the ingress point. This was replaced with Kong Ingress Controller because Kong provides JWT validation, CORS, and rate limiting as declarative plugins on Kubernetes CRDs, removing the need to duplicate those concerns in each FastAPI service. The NGINX deployment was explicitly decommissioned after Kong reached parity.

GitOps over SSH deploy: The initial delivery plan used SSH-based docker compose deployment. This was replaced with a full GitOps stack (ArgoCD + k3s + Kustomize) during Sprint 3. Although this added unplanned work, it eliminated SSH key management in CI, provided self-healing, drift detection, and a git-native deployment audit trail.

Database-per-service: Each microservice owns a dedicated PostgreSQL instance. This enforces bounded contexts, prevents tight coupling at the data layer, and allows each service to evolve its schema independently. Kafka is the only cross-service communication channel.

Stage 3 multi-tenancy: The forward plan provisions one isolated k3s cluster per customer via Terraform (KAN-61), explicitly rejecting namespace-based multi-tenancy to guarantee blast-radius isolation between customers.

Observability and Resilience

Each FastAPI service exposes a Prometheus metrics endpoint via prometheus-fastapi-instrumentator. A Grafana dashboard tracks request rate, error rate, and latency per service. SLIs and SLOs are formally defined (e.g. vote latency under 500ms, 99% uptime), and Prometheus Alertmanager fires a service-down alert when a pod becomes unavailable.

Structured JSON logging is applied across all services using Python logging or loguru, making log aggregation and querying straightforward. Fault tolerance was validated by killing services mid-flow: Kafka replays buffered events on audit service restart, the gateway returns clean errors when the voting service is down, and Kubernetes restarts pods automatically via liveness probes.

Boundaries and Scope

The MVP targets small to medium organisations requiring a self-hosted, auditable voting system. The platform is not designed for public-internet elections at scale; it assumes a known voter population registered ahead of the election. Voter anonymity is achieved through the separation between authentication (who voted) and the ballot record (what was voted), stored in separate services with separate databases.

Namespace-based multi-tenancy is explicitly out of scope. Each customer deployment is an isolated k3s cluster provisioned by Terraform, ensuring no shared kernel namespaces or network paths between tenants. The Stage 3 admin portal (KAN-62) will expose a company-only UI to trigger provisioning jobs, abstracting Terraform execution behind an API.
